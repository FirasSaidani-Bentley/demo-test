1
Introduction
This guide supports participants of the internal AI hackathon. Its goal is to help every team, regardless of technical
background, deliver a working prototype by the end of the event. The selected tool is GitHub Copilot in VS Code,
for three reasons. It is natively integrated into the development environment. It offers a choice of multiple AI
models depending on the task.
It supports customization through versioned files, which makes sharing team
conventions easy.
The guide reads linearly for a first introduction, or section by section to address a specific need during the
hackathon.
ò Prerequisites
VS Code installed. Active GitHub Copilot subscription on your Bentley account. Git configured locally.
2
The four Copilot chat modes
Copilot offers four operating modes, accessible from the dropdown at the top of the Chat view. The choice of
mode determines what Copilot can do. Using the wrong mode is the most common source of frustration at the
start of a hackathon.
2.1 Ask mode
Pure conversation. Copilot answers questions and suggests code for you to copy manually. No files are modified.
Useful to understand a concept, clarify an error, or get a quick snippet.
2.2 Edit mode
Guided modification. You select one or several files, describe the intended change, Copilot proposes edits as a diff
that you accept or reject. Useful for refactoring, adding error handling, translating comments.
2.3 Agent mode
Full autonomy.
You give a high-level objective, Copilot plans, creates files, runs terminal commands, installs
dependencies, and iterates until the task is complete. This is the recommended mode for scaffolding an app or
adding a significant feature.
2.4 Plan mode
Planning before action. Copilot drafts a detailed plan of what it will do before executing. You review the plan,
then Copilot applies it. Useful for complex tasks where you want to avoid the agent going in the wrong direction.
 Practical rule
For a hackathon, use Agent to build, Edit to refine, and Ask to debug.
3
AI model selection
At the bottom of the Chat view, a selector lets you change the model used by Copilot. The main available models
are Claude Sonnet 4.6, Claude Opus 4.5, GPT-5.2, Gemini 2.5 Pro, and Grok Code Fast 1.

6
How the files talk to each other
The power of the setup comes from the fact that agents write to shared files, and read the files written by other
agents. The files are the memory of the team.
Here is what happens when you select builder and ask it to implement a feature:
1. Copilot loads copilot-instructions.md. The agent now knows the project context, the Bentley visual tokens,
and the writing conventions.
2. Copilot loads builder.agent.md. The agent now takes on the builder persona.
3. The mandatory context load kicks in. Copilot reads strategy/PROBLEM.md, strategy/VALUE.md, docs/ARCHITECTURE
and pitch/NARRATIVE.md.
4. Only then does Copilot look at your actual prompt.
The same happens when you switch to pitch-director. The agent reads the strategy written by strategist
and the architecture written by architect. The narrative it produces is grounded in the real project, not in
assumptions.
 The key insight
Agents never talk to each other directly. They talk through the Markdown files. When strategist updates
PROBLEM.md, every subsequent call to any agent inherits that update automatically. The team stays aligned
without any manual coordination.
7
Shortcut prompts
Beyond agents, the setup includes five shortcut prompts stored in .github/prompts/. Unlike agents, which are
personas you switch into, shortcut prompts are atomic actions you invoke anytime by typing a slash command in
the chat.
7.1 What they are
A prompt file is a short Markdown file with a YAML header that declares a description, and a body that instructs
Copilot to do one specific task. When you type the slash command, Copilot loads the prompt body on top of the
current context and executes it.

7.2 When to use shortcuts versus agents
A simple rule of thumb:
• Use an agent when you need a specialist to take on a role across a phase of work (framing the strategy, designing
the UI, building the pitch)
• Use a shortcut prompt when you need a one-shot action that can be described in a few lines and invoked
many times during the day
Agents are your team. Shortcuts are your keyboard.
 Create your own shortcuts
If your team finds itself repeating the same request three or four times during the day, turn it into a prompt.
Create a Markdown file in .github/prompts/ with a YAML header and a body, commit it, and the shortcut
becomes available to everyone on the team. The setup grows with the project.
8
Keeping the project consistent
As the day progresses, different agents update different files. The strategist refines PROBLEM.md. The architect
locks choices in ARCHITECTURE.md. The feature-designer reprioritizes ROADMAP.md. Each change is local, but
together they can drift out of sync.
The consistency-keeper agent exists to solve exactly this problem.
8.1 What it does
Unlike the other agents, consistency-keeper does not produce new content. It reads every file in strategy/,
docs/, and pitch/, and reports what no longer lines up. For example:
• A feature was removed from ROADMAP.md but is still described in DEMO_SCRIPT.md
• The target user changed in PROBLEM.md but NARRATIVE.md still references the old persona
• ARCHITECTURE.md mentions a backend, but the actual code in src/ does not have one
• VALUE.md claims a metric the prototype cannot actually demonstrate
The agent flags each issue with a file reference and a suggested fix. You decide what to apply.

8.3 When to call it
Three natural moments during a hackathon day:
1. Mid-day reset. Around lunch, call consistency-keeper to catch drift before it compounds.
2. Before the security audit. Clean inconsistencies first, so the auditor works on a coherent project.
3. Before the pitch preparation. The pitch-director produces stronger narratives when the underlying files
are aligned.
 Why this agent exists
Without it, inconsistencies surface during the pitch Q&A, when an executive asks a sharp question and the
team realizes the demo contradicts the narrative. With it, those contradictions are caught and fixed hours
earlier, when the cost of fixing them is trivial.
9
Editing the files yourself
Every file in this setup is plain Markdown. You open it, you read it, you edit it with your keyboard. Nothing is
hidden in a database or a special tool.
• If an agent produced something slightly off, open the file and fix the phrasing yourself
• If you want an agent to behave differently, open its .agent.md and adjust the instructions
• If your project needs a new strategic file, create a Markdown file in strategy/ and reference it in copilot-instructions
The setup is not a black box. It is a set of readable files that you and your agents share. The more you read
them, the more you understand what the team is doing.
